Distributed Systems and Vector Search: Architecture, Scalability, and Modern Retrieval Techniques

Modern distributed systems must balance consistency, availability, and partition tolerance — a tradeoff formalized by the CAP theorem. In practice, most large-scale deployments choose eventual consistency over strict ACID guarantees to achieve the throughput and latency profiles demanded by real-world traffic. Systems like Apache Cassandra and DynamoDB are designed around this tradeoff, using consistent hashing to distribute data across nodes and gossip protocols to propagate state changes without a central coordinator. The consequence is that reads may return stale data for a window of time, which applications must be designed to tolerate.

Handling peak load is one of the most critical operational challenges in distributed infrastructure. When traffic spikes — due to product launches, marketing events, or denial-of-service attacks — systems that rely solely on vertical scaling will exhaust available compute resources and begin dropping requests. Horizontal scaling, where additional service replicas are spun up in response to demand, is the standard mitigation. Kubernetes addresses this through the Horizontal Pod Autoscaler, which monitors custom metrics such as CPU utilization, memory pressure, or queue depth and adjusts replica counts accordingly. However, scaling is not instantaneous; there is a lag between the metric threshold being crossed and new pods becoming ready. To absorb this lag, systems use a combination of rate limiting at the API gateway layer, circuit breakers to prevent cascade failures, and request queuing with back-pressure signals to upstream producers.

Load balancers distribute incoming requests across available replicas using algorithms such as round-robin, least-connections, or consistent hashing. In session-aware applications, consistent hashing ensures a given client is always routed to the same backend, preserving in-memory state without external session storage. Health checks are performed at configurable intervals; unhealthy replicas are removed from the load balancer pool automatically. Blue-green and canary deployment strategies leverage load balancers to shift traffic gradually between old and new versions, enabling rollbacks within seconds if error rates spike.

Capacity planning requires understanding the relationship between throughput, latency, and resource utilization. The Universal Scalability Law models the degradation in throughput as concurrency increases, capturing both the overhead of coordination and the effects of serialization bottlenecks. Engineers use load testing tools such as Locust, k6, or Gatling to drive synthetic traffic at multiples of expected peak load, measuring p50, p95, and p99 latency percentiles. The inflection point — where latency begins increasing faster than throughput — defines the practical capacity ceiling. Beyond this point, queuing theory predicts latency growth will become superlinear.

Data consistency during high-throughput ingestion pipelines requires careful coordination. Write-ahead logs ensure durability by persisting operations to disk before acknowledging them to the caller; if the process crashes mid-write, the log is replayed on restart. Two-phase commit provides atomic cross-node writes but introduces a blocking protocol that reduces availability during coordinator failures. The saga pattern decomposes a distributed transaction into a sequence of local transactions with compensating rollback actions, achieving eventual consistency without a global lock. Conflict-free replicated data types allow concurrent updates across replicas without coordination, resolving conflicts deterministically using merge semantics.

Idempotency is a critical property for ingestion pipelines that process messages from queues such as Apache Kafka or Google Pub/Sub. Because at-least-once delivery guarantees allow duplicate messages, each write operation must produce the same result regardless of how many times it is applied. This is typically achieved by including a unique message identifier in each write and using conditional updates that reject writes where the identifier already exists. Change data capture systems, which stream database mutations to downstream consumers, also require idempotent consumers to handle the redelivery of events after consumer restarts.

Embedding models transform raw text into dense numerical vectors that encode semantic meaning in a continuous high-dimensional space. The transformer architecture, introduced in the paper Attention Is All You Need, uses self-attention mechanisms to build context-aware representations of each token in a sequence. Sentence-level embeddings are derived by pooling token representations — typically using mean pooling or the CLS token — producing a single fixed-dimensional vector per input sequence. Models such as all-MiniLM-L6-v2, E5, and GTE are optimised for semantic similarity tasks through contrastive learning objectives that pull embeddings of similar sentences together and push dissimilar ones apart in the vector space.

The quality of embeddings for retrieval tasks depends heavily on the domain and vocabulary of the corpus. General-purpose models trained on web text may fail to capture the nuances of specialised domains such as legal, medical, or financial text. Fine-tuning embedding models on domain-specific sentence pairs using a contrastive loss such as Multiple Negatives Ranking Loss significantly improves retrieval recall in these settings. Evaluation is performed using benchmark datasets such as BEIR, which contains retrieval tasks across diverse domains including scientific papers, news articles, and question-answering corpora.

Vector databases store high-dimensional embeddings and support approximate nearest neighbour queries with sub-millisecond latency at billion-vector scale. The core algorithmic challenge is trading off recall — the fraction of true nearest neighbours returned — against query latency. Flat indices perform exact nearest neighbour search by comparing the query vector against every stored vector; this guarantees perfect recall but scales linearly with corpus size, making it impractical above a few million vectors. Approximate methods such as HNSW construct a navigable small-world graph where edges connect vectors that are geometrically close; queries traverse the graph greedily from a random entry point, pruning branches that diverge from the query vector.

FAISS, developed by Meta AI, is the most widely used open-source library for dense vector search. It supports both exact and approximate indices, GPU acceleration, and a range of quantization techniques for compressing vectors. IndexFlatIP performs exact inner-product search; when vectors are L2-normalised before insertion, inner product equals cosine similarity, making it equivalent to IndexFlatL2 on unit vectors. The IVF family of indices partitions the vector space into Voronoi cells using k-means clustering; queries search only the cells nearest to the query vector, dramatically reducing the number of comparisons at the cost of some recall. Product quantization further compresses vectors by decomposing them into subvectors and encoding each subvector using a shared codebook, reducing memory requirements by an order of magnitude.

Cosine similarity measures the angle between two vectors, ignoring their magnitude, and is the preferred metric for language embeddings because semantic relatedness is encoded in the direction of the vector, not its length. Two sentences with the same meaning but different lengths will have embeddings pointing in the same direction but potentially at different magnitudes; cosine similarity correctly identifies them as similar while Euclidean distance would penalise the magnitude difference. Euclidean distance is appropriate when the magnitude of the embedding vector encodes meaningful information, as in face verification systems trained with metric learning objectives that calibrate L2 distance as a confidence score.

Retrieval-Augmented Generation combines a retrieval component with a generative language model to produce grounded, factual responses. At query time, the user's question is embedded and used to retrieve the most semantically similar document chunks from a vector store. These chunks are prepended to the language model's input as additional context, enabling it to answer questions about information not present in its training data. This approach significantly reduces hallucinations compared to purely parametric generation because the model can cite specific retrieved passages rather than interpolating from memorised training distributions. Chunking strategy has a significant effect on retrieval quality; chunks that are too small lose context, while chunks that are too large dilute the relevance signal with off-topic content.

Query expansion improves retrieval recall by augmenting the user's query with semantically related terms before embedding. Pseudo-relevance feedback uses the top-k retrieved documents from an initial search to identify expansion terms. HyDE, or Hypothetical Document Embedding, prompts a language model to generate a hypothetical answer to the query and embeds that answer rather than the original question; because the hypothetical answer resembles a relevant document more than the question does, it typically retrieves higher-quality results. LLM-based query rewriting reformulates ambiguous or underspecified queries into more explicit forms that capture the user's intent more precisely.

Vertex AI Vector Search, formerly known as Matching Engine, is Google Cloud's managed approximate nearest neighbour service. It ingests embedding data from Cloud Storage in JSON Lines format, where each line contains a vector and its associated metadata. The service builds either a tree-AH or HNSW index depending on the configured algorithm and exposes gRPC and REST endpoints for querying. Deployed index endpoints support both batch and streaming updates; streaming updates allow individual vectors to be inserted or deleted without rebuilding the entire index, which is critical for corpora that change frequently. The service scales automatically to handle query throughput and can serve queries with single-digit millisecond latency at billion-vector scale when configured with dedicated resources.

Migrating a local FAISS-based RAG system to Vertex AI Vector Search involves exporting embeddings to Cloud Storage, creating an index with the appropriate algorithm and distance metric configuration, deploying the index to an endpoint, and replacing the local search call with the Vertex AI SDK match method. The embedding model can be swapped from a local sentence-transformers model to the Vertex AI textembedding-gecko model, which is hosted as a managed API endpoint and requires no infrastructure management. For query expansion, the local mock can be replaced with a call to the Gemini API via the Vertex AI generative models SDK. The overall pipeline architecture remains unchanged; only the implementations of the embedder, vector store, and expander interfaces need to be swapped.
